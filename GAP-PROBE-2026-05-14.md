# P3 gap-fill experiment — gap is not fillable
*Run date: 2026-05-14. Closes the polish-queue P3 item.*

## Question

Polish-queue P3: "K.2862's three known siblings (K.2868, K.5065.A, Rm.111)
have transliteration content at eBL but are missing from `/fragments/all-signs`
and have empty `lineToVec` on `/fragments/<id>`. eBL stats: 36,583
transliterated, 36,493 in all-signs, 36,328 in our cache with non-empty
`lineToVec` — gap of ~250 fragments, concentrated in joined ones. Pass would
crawl every museum number declared in any cached fragment's `joins[]`; most
will return empty, but some will fill the gap. Cost: ~1,000 extra HTTPs,
~3 min."

The assumption: at least *some* of the ~250 gap fragments would return
usable signs content via `/fragments/<id>` even though they don't appear
in `/fragments/all-signs`.

## Method

1. Pick 300 random K.* fragments from the corpus (K. = Kuyunjik, the most
   heavily-joined collection — best signal-to-noise for sampling the gap).
2. Fetch each via `/api/fragments/<id>` and parse `joins[]`. Note: each
   join entry's `museumNumber` is an *object* `{prefix, number, suffix}`,
   not a string — earlier sweep gave false-zero results because of this.
3. Collect every museum-number in any join group that is NOT present in
   `all-signs-full.json` (i.e. the in-corpus index). This is the gap set
   reachable from the 300-fragment sample.
4. Probe every gap candidate with `/api/fragments/<id>` and count how many
   return non-empty `signs` content, non-empty `lineToVec`, or empty.

## Result

| metric | value |
|---|---|
| K.* fragments in corpus | 12,678 |
| K.* fragments sampled | 300 |
| Sampled fragments with ≥1 gap-member in joins[] | **46 / 300 (15%)** |
| Unique gap candidates discovered | **106** |
| Gap candidates with non-empty `signs` content | **0 / 106 (0%)** |
| Gap candidates with non-empty `lineToVec` only | 0 / 106 |
| Gap candidates with empty payload (200 OK, no content) | 106 / 106 |
| Errors during probe | 0 |

Every probed gap candidate returns 200 OK from `/fragments/<id>` — these
museum numbers exist in eBL's database — but every one returns empty
`signs` and empty `lineToVec`. eBL knows the fragment exists (because some
other in-corpus fragment claims a join to it) but has no transliteration
content recorded for it yet.

This pattern matches the K.2862 → K.2868/K.5065.A/Rm.111 case exactly, and
generalizes: **the gap is a function of eBL's transliteration completeness,
not a `/fragments/all-signs` omission.**

## Why this is a stronger result than "P3 doesn't lift recall"

If even one gap candidate had returned non-empty signs, P3 would have been
viable (or at least worth a full crawl). The 0/106 result is monotonic — at
this rate the expected number of usable fragments in the full ~250 gap is
< 3, and even those would have minimal corpus impact since the validation
baseline doesn't include any gap members as targets or siblings.

The bug-fix in step 2 above is also load-bearing. My first sweep returned
"0 gap candidates from 300 fragments" because I treated `joins[i].museumNumber`
as a string instead of an object. That false-zero almost concealed the
real finding: 46/300 fragments DO have gap-joins, the gap is real and
common, the fragments just have no content.

## Decision

**P3 closed as a no-op.** The gap exists and is approximately as large as
the polish-queue estimated (~250 fragments), but eBL has zero signs/lineToVec
content for any of them. No client-side crawl can fill what the server
doesn't have.

What this *does* unlock: surfacing "known sibling, no transliteration
available at eBL" annotations in `find_parallel_text` output when the
target has join entries pointing at gap fragments. That's an informational
UX enhancement, not a matcher improvement — out of polish-queue scope but
worth noting for a future feature pass.
