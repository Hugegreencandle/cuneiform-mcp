# v0.22 — `build_canonical_recension_tree` design note

**Status:** v0.22 watershed tool. Companion to `build_scribal_school_graph` (built in a parallel worktree).

**Goal:** Given a single tablet (`seed_tablet_id`) known to be a manuscript of a multi-witness composition, automatically reconstruct the **textual family tree** (stemma) of all extant witnesses by clustering on shared chunk-hash signal, then running phylogenetic tree inference. This is *the* classic Assyriological problem — scholars currently do it by hand over weeks per composition (cf. Walker & Dick, *The Induction of the Cult Image*, SAALT 1, 2001, esp. §1.2 on the Mīs pî manuscript tradition).

The corpus-wide chunk-hash index shipped in v0.20 (96,654 non-singleton length-20 windows over the eBL corpus) supplies the inter-witness signal; this tool consumes it.

---

## 1. Algorithm choice — neighbor-joining (NJ) by default

We evaluated four candidates:

| Algorithm | Input | Output | Cost | Assumes | Verdict |
|---|---|---|---|---|---|
| **Neighbor-joining (NJ)** | distance matrix | unrooted binary tree | O(N³) | additivity | **DEFAULT** — robust, no clock |
| UPGMA | distance matrix | rooted, ultrametric tree | O(N²) | molecular clock | Secondary option (when caller wants a rooted tree) |
| Maximum parsimony | character matrix | tree(s) | NP-hard | discrete characters | Out of scope: chunk-hash signal isn't a clean character matrix |
| Bayesian (MrBayes-style) | characters + prior | posterior tree dist. | hours | substitution model | Out of scope: no defensible substitution model for cuneiform |

**Why NJ as default.** Textual phylogenetics (cf. Mooney et al. on Greek NT manuscripts, the *Canterbury Tales* phylomemetic project) has converged on distance-based methods for two reasons: (1) scribal transmission is patently *not* clock-like — a Middle-Babylonian manuscript four generations downstream of a Neo-Assyrian autograph can be more textually conservative than a Late-Babylonian sister of that autograph — so UPGMA's ultrametric assumption is wrong on its face; (2) hand-built stemmata in Assyriology are themselves unrooted in practice (Walker & Dick draw Mīs pî connections as a network, not a rooted tree). NJ's unrooted output matches the field's intuition.

UPGMA is offered as a secondary `algorithm` option because some users want a rooted tree for display purposes (with branch lengths interpretable as time-from-common-ancestor *if* you accept the clock assumption). We never recommend it for inference, only display.

**Rooting.** NJ produces an unrooted tree of N+(N−2) nodes connected by 2N−3 branches. We expose the unrooted tree in Newick using a trifurcation at an arbitrary internal node (the last node created during NJ). Callers wanting a midpoint root can post-process; we do not impose one because midpoint rooting silently encodes a clock assumption.

## 2. Distance metric — chunk-overlap with `max()` denominator

For two tablets A, B with chunk-host sets HA, HB and the shared-chunk count S(A,B):

```
distance(A, B) = 1 − S(A,B) / max(|HA|, |HB|)
```

Range: [0, 1]. 0 = every chunk of the smaller tablet appears in the larger (perfect textual containment). 1 = no shared chunks.

**Why `max()` and not Jaccard `|HA ∪ HB|`?** Cuneiform manuscripts of the same composition vary wildly in extant length: K.5896 has 303 indexed chunks; its sister K.9508 has 65 (it's a fragment). With Jaccard:

```
J = 1 − 65 / (303 + 65 − 65) = 1 − 65 / 303 ≈ 0.785
```

…which is exactly what max() gives here, but as soon as HA and HB diverge in shared content the two diverge. The *union* denominator penalizes A's unshared chunks (signs that appear in A but not in B) as if they were B's failure to be similar. For a fragment B that is *fully contained* in A, we want distance to reflect "B has been completely reproduced in A," not "A has signs B lacks." The `max()` denominator gives `1 − |HB|/|HA|` when B ⊂ A, which is the smallest distance achievable for a fragment of that relative size.

**Why not `min()` denominator?** It would give 0 for any small fragment fully contained in any other tablet, regardless of how much else A holds — too generous. `max()` is the conservative middle path.

**Why not sum?** Sum denominators (the classic Sørensen–Dice in disguise) symmetrize but harshly penalize length asymmetry. We tested it informally against K.5896's neighborhood and `max()` produced tighter clusters around true Mīs pî witnesses.

**Trade-off / limitation.** Two tablets A and B might share many chunks because they're both copies of *Tablet III* of a multi-tablet series and the chunk index doesn't know about composition boundaries. We accept this — the seed_tablet_id implicitly defines the composition. If the seed is K.5896 (Mīs pî Tablet I in Walker–Dick), the BFS-expanded witness set will be dominated by other Mīs pî witnesses.

## 3. Witness enumeration — single-hop BFS from seed

```
1. seed_chunks = getChunksContaining(seed_tablet_id)
2. For every other tablet T that appears as an occurrence in seed_chunks,
   count its shared chunks with the seed.
3. Keep T iff shared(seed, T) >= min_pairwise_chunks.
4. Sort by shared-count desc, cap at max_witnesses.
5. Add the seed itself to the set.
```

**Why single-hop, not transitive closure?** A second-hop expansion (witnesses of witnesses) would drag in tablets that share content with a witness *T* but not with the seed — e.g., if T is a commentary on the composition AND quotes another work, those quoted works pollute the cluster. Single-hop keeps the cluster anchored to the seed's textual identity.

**The `min_pairwise_chunks` parameter.** Default 3. One shared length-20 chunk can be coincidental (the corpus has cross-curricular formulae — see find_incipits, find_formulaic_passages). Three is the empirically-calibrated threshold where the K.5896 neighborhood becomes the Mīs pî tradition and nothing else (8 witnesses at default cap). Lower to 2 for more lacunose manuscript clusters; raise to 5 for highest-confidence sister identification.

**The `max_witnesses` cap (default 50).** NJ is O(N³). At N=50, the matrix is 50×50 = 2500 cells, NJ runs in ~125,000 ops — trivial. At N=200, 8M ops, still fine. We cap at 50 because beyond that, the tree becomes visually unwieldy and the distance signal degrades (witnesses with <3 shared chunks are noise).

## 4. Output — Newick + edge list

Newick is the standard text format for phylogenetic trees, parseable by FigTree, iTOL, ETE3, Bio.Phylo, etc.:

```
(A:0.1,B:0.2,(C:0.3,D:0.4):0.5);
```

For NJ output, we emit an **unrooted** Newick string with a trifurcation at the root. Branch lengths are the NJ-computed distances from each node to its parent (can be negative in NJ — we clamp to 0 with a warning, standard practice).

We also emit `tree_edges` as a flat `Array<{from, to, branch_length}>` for programmatic consumption — e.g., downstream tools that want to traverse the tree without writing a Newick parser.

Internal nodes are labelled `N1, N2, …` in NJ creation order. The final unjoined pair, when we reach N=2 remaining taxa, is joined with a final branch — that branch's midpoint is conventionally the root display anchor, but we do not enforce it.

## 5. Limitations

1. **Exact-chunk-share ≠ descent.** Two manuscripts sharing many chunks could be (a) parent and child, (b) sibling copies of a common ancestor, or (c) independent copies of a fixed scholastic stream. The chunk index cannot distinguish these. A phylogeneticist would say: NJ recovers the *unrooted topology* but not the direction of descent.
2. **Rooted vs. unrooted.** As above. We do not root.
3. **No molecular clock.** Branch lengths are textual distances, not time. Two tablets a thousand years apart can be textually closer than two tablets a century apart if the older tradition was held conservative.
4. **Composition boundary.** The seed-defined cluster captures whatever the seed shares chunks with — if the seed is a multi-tablet series, the cluster may include sister tablets *and* parallel tablets of the same series.
5. **NJ negative branch lengths.** Saitou–Nei NJ can produce negative branches when the input distance matrix violates additivity. We clamp to 0 and emit a warning. This is standard (cf. Felsenstein, *Inferring Phylogenies*, 2004, §11.4).
6. **No bootstrap.** A proper phylogenetic study would resample the chunk-hash signal to compute branch-support values. Out of scope for v0.22; queued for v0.23 (see Future work below).

## 6. Benchmark target — Mīs pî / K.5896

Walker & Dick (2001) hand-reconstructed the manuscript family of *Mīs pî*. The methods paper §3.7.3 already references this composition (K.9508 reproduces a 142-sign chain from K.5896, documented in the v0.19 audit). Our automated reconstruction of the K.5896 cluster is a candidate gold-standard benchmark — if our NJ output identifies K.9508 + K.6683 + K.15325 + K.8994 as the inner Mīs pî family, and Walker–Dick concur, we have a reproducible automated stemma.

The Round-7 audit (this commit) verifies the algorithm at the structural level: the tree is well-formed Newick, the distance matrix is valid, and K.9508 sits in the top-3 closest witnesses to K.5896 — call this the *adjacency benchmark*. A full topology benchmark against Walker–Dick is deferred to a separate methods-paper amendment.

## 7. Future work

- **Bootstrap support values.** Resample chunk-hash signal (e.g., 100 replicates dropping 10% of chunks) and re-run NJ; report branch-frequencies. Standard phylogenetics practice.
- **Rooted-tree inference.** Add an `outgroup_tablet_id` parameter for caller-supplied outgroup rooting (canonical phylogenetic rooting strategy).
- **Bayesian inference.** If we ever define a defensible scribal substitution model (e.g., sign-pair confusion rates from the orthographic-outliers tool), MrBayes-style inference becomes possible.
- **Topology benchmark.** Curate ~5 hand-stemmatized compositions (Mīs pî, Šurpu, Anzû, Erra, Maqlû) as a gold-standard set and report Robinson–Foulds distance between automated and scholarly trees.
