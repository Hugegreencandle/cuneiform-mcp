# cuneiform-mcp — Cloudflare Workers Hosting Plan (v1.0 territory)

**Status:** Design recon for review. Not implementation-ready — this is the architecture-decisions doc that a v1.0+ implementation plan would build on. Roughly aligned with `docs/post-v0.20-roadmap.md` Tier 3 polish + the v1.0 API-freeze milestone.

## Context — why hosting, why now

Currently cuneiform-mcp is a **local-only MCP server**. Every researcher who wants to use the tools must:
1. Clone the repo
2. `npm install` + `npm run build`
3. Run `node scripts/build-signs-index.mjs` + `build-chunk-index.mjs` + `build-incipits-index.mjs` (~1–2 hours of cache build, plus an enrichment burst on top)
4. Wire the local `dist/index.js` into their Claude Code MCP config

That's a substantial onboarding cost for someone who just wants to run `find_formulaic_passages` once to see what shows up. The methods-paper acceptance path (CDLJ rejected, JOHD next, Zenodo permanent) naturally produces public-readiness pressure — *if* the paper lands, peer researchers will want to query the results without becoming MCP-server operators themselves.

The Cloudflare Workers ecosystem now ships first-class support for **MCP-over-HTTP** (the [streamable HTTP transport](https://modelcontextprotocol.io/specification/draft/basic/transports#streamable-http), MCP spec rev 2025-06-18+) — which is what makes this a real plan rather than just an idea. We can serve the existing 68-tool MCP surface unmodified, behind a public URL, against pre-built read-only indexes.

This is **not** about rate-limit-evasion for eBL — the rate-limit discussion (`reference_xrpl_*` notwithstanding) is a separate concern handled through Prof. Jiménez. This plan is purely about serving *our* derived data to other researchers.

## Proposed architecture

```
                       ┌─────────────────────────────────────────┐
                       │  Cloudflare Worker (cuneiform-mcp.dev)  │
                       │  • streamable HTTP MCP transport        │
                       │  • tool dispatch (unchanged from local) │
                       │  • cached at edge per geo               │
                       └────────────┬────────────────────────────┘
                                    │ R2 reads
                                    ▼
                       ┌─────────────────────────────────────────┐
                       │  Cloudflare R2 bucket                   │
                       │  • all-signs-full.json       (~33 MB)   │
                       │  • fragment-metadata.json    (~50 MB)   │
                       │  • chunk-index.json          (~67 MB)   │
                       │  • incipits-index.json       (~94 MB)   │
                       │  • anomaly-index.json        (~14 MB)   │
                       │  • signs-trigram-index.json  (~10 MB)   │
                       │  • lineToVec-corpus.json     (~7 MB)    │
                       │  • semantic-embeddings.bin   (~50 MB)   │
                       │  • Total: ~325 MB                       │
                       └─────────────────────────────────────────┘
                                    ▲
                                    │ batch write (Dane laptop → R2)
                                    │ per-release; CI optional
                       ┌────────────┴────────────────────────────┐
                       │  Local build pipeline (unchanged)       │
                       │  • scripts/build-*.mjs                  │
                       │  • scripts/enrich-burst-*.mjs           │
                       │  • Sync to R2 on tag push (v0.X.0)      │

```

The Worker is a thin dispatcher. All 68 tools' core logic is unchanged — same `src/*.ts` modules, just loaded by a different entry point (`worker.ts` instead of `index.ts`).

## Decisions to lock before building

### 1. Transport: streamable HTTP MCP (recommended)

The streamable HTTP transport is what Anthropic, Cloudflare, and others have converged on for hosted MCP servers. Per the [MCP spec rev 2025-06-18+](https://modelcontextprotocol.io/specification/draft/basic/transports#streamable-http), clients POST to a `/mcp` endpoint and receive streaming responses. Existing `dist/index.ts` uses the stdio transport; the Worker entry point swaps to the HTTP transport.

**Recommendation:** ship HTTP-only on the hosted version. Local users keep stdio. Don't try to support both transports in one binary — the Worker stays minimal, the local CLI stays familiar.

### 2. Auth: public read, no auth (recommended)

The derived data is academic-research output we *want* to be readable. No login, no tokens. Rate limiting via Cloudflare's built-in DDoS / abuse controls (or a soft per-IP cap at the Worker if needed).

**Recommendation:** public read. Anyone with the URL can hit `find_formulaic_passages`. The methods-paper-acceptance value of "no setup, just paste the MCP URL into Claude Code" outweighs the abuse risk.

If write-path access is ever needed (for cache refresh), gate it with a Cloudflare Access bearer token — only Dane's machine sends it.

### 3. Update cadence: tag-aligned, manual trigger (recommended)

Every release (v0.23, v0.24, ...) syncs the local caches to R2 as a tagged versioned snapshot:
- `r2://cuneiform-mcp/v0.22.0/chunk-index.json`
- `r2://cuneiform-mcp/v0.22.0/fragment-metadata.json`
- ... etc.
- `r2://cuneiform-mcp/latest/...` symlink-equivalent (R2 doesn't have symlinks; use a manifest file pointing at the current tag)

The Worker reads the `latest` manifest at startup, then pins to that version for the request lifetime. Clients can request a specific version via header (`X-Cuneiform-Version: 0.22.0`) for reproducibility.

**Recommendation:** manual sync trigger for now (`scripts/sync-caches-to-r2.mjs`), wire to GitHub Actions later if the cadence becomes regular.

### 4. Versioning at the API level

Tool signatures evolve (we added `host_genres_spanned` in v0.19.1, two new tools in v0.21, two more in v0.22). For the hosted API, that's a stability concern: a client built against v0.22 shouldn't break when v0.23 ships.

**Recommendation:** the hosted MCP server's tool list is "what the worker code knows about" — same versioning as the local binary. v1.0 freezes the tool *names* and required-field shapes; subsequent versions can only add optional fields or new tools. The chunk-index data format is separate from the tool API and can evolve more freely behind the versioned R2 manifests.

### 5. Costs (rough estimate)

Cloudflare Workers free tier: 100k requests/day. Paid: $5/month base + $0.30 per million additional requests. R2: $0.015/GB-month storage (~$5/year for ~325 MB) + $0.36 per million reads (free for the first 10 million/month).

For a methods-paper-publication audience size, the **expected monthly cost is well under $10**, dominated by the Workers Paid plan baseline. Doesn't move the needle.

### 6. Migration path for existing users

None required. Existing local-stdio users (currently: Dane + Quigles1337 read-only) keep their local installations and their `~/.cache/cuneiform-mcp/` builds. The hosted instance is a parallel deployment, not a replacement.

If the local user wants to switch to hosted, the migration is: remove the local MCP server registration from Claude Code config, replace with `{ "type": "http", "url": "https://cuneiform-mcp.dev/mcp" }`. That's it. No data migration, no auth setup.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| MCP streamable HTTP transport stability — relatively new (mid-2025), some client behavior still settling | Ship on the *current* spec; pin Worker to a known SDK version; revisit if the spec changes incompatibly. |
| Stale R2 caches drift from upstream eBL | Each cache file carries a `_meta.build_timestamp`. Worker surfaces this in tool responses (`provenance.fetched_at` already does for some tools). Researchers can see how stale the data is. |
| Provenance / attribution to eBL | Worker prepends a one-line eBL attribution to every tool response (in the rendered text section). Methods paper §3 already credits eBL; the hosted instance carries the same. |
| Abuse: a researcher / agent burning through Worker quota | Cloudflare's built-in rate-limiting at the worker level. If abuse becomes real, add per-IP cap. |
| Cost spike from a popular paper citation | Monitor monthly; the cost is so low that even a 10× spike is sub-$100/month. Real cap if needed. |
| Compromise of derived data | The R2 bucket holds *derived* caches, not personal data. Worst case: someone copies our chunk-index. Not really a compromise — that's basically the goal. |
| eBL changes API shape and breaks our caches | Caches are versioned. Worker pins to a known version. Local build pipeline regenerates from latest eBL on next release; hosted updates on next sync. |

## Out of scope (deliberate)

- **Hosted MCP server for *consuming* eBL in real-time.** That would route every user's query through eBL on our infrastructure, defeating the whole point of caching and putting us back in the rate-limit-discussion territory. Hosted version serves *only* pre-built derived data.
- **Write-path tools.** All 68 tools are read-only (queries against pre-built indexes). If we ever ship a write-path tool (e.g. "submit corrections to fragment-metadata"), that's a separate auth + storage problem.
- **Multi-tenancy / per-researcher namespaces.** Single shared instance. If a researcher needs their own derived caches, they run locally.
- **GPU/ML inference at the Worker.** v0.23's `sign2vec` (post-v0.20-roadmap.md item #2) might want an inference path; if so, that's a separate Cloudflare AI Workers integration question.

## Sequencing — when to build

Three release gates:

| Gate | When | What |
|---|---|---|
| **Trigger 1** (start design) | Methods paper acceptance signal (JOHD or fallback venue) | Validate the demand: is there a researcher who actually wants this? Don't pre-build without that signal. |
| **Trigger 2** (start build) | First external researcher requests access | Build the minimum viable Worker + R2 sync in 1-2 weeks of focused work. Doesn't need to be feature-complete — just enough to serve the 68 tools against the v1.0 caches. |
| **Trigger 3** (public launch) | After v1.0 (per `docs/post-v0.20-roadmap.md`) | API frozen, caches stable, methods paper published or accepted. Then announce. |

**Recommendation:** **don't build until Trigger 1.** Building before the paper is accepted is premature — the hosted layer is value-add for paper readers, not value-add for the development workflow.

## What the implementation plan would look like (deferred to actual build time)

If/when Trigger 1 fires:

1. `worker.ts` — Cloudflare Worker entry point that loads the same `src/*.ts` modules and serves MCP over HTTP. ~150 LOC.
2. `scripts/sync-caches-to-r2.mjs` — push local `~/.cache/cuneiform-mcp/*.json` to versioned R2 paths. ~80 LOC.
3. `worker-config.toml` — wrangler config: R2 binding, secrets, routes.
4. `docs/hosted-deployment.md` — operational runbook (deploy, rotate, monitor).
5. A v1.0 milestone audit verifying the hosted instance matches local-stdio behavior on a fixed test query set.

Approximate effort: 1-2 weeks of focused work post-trigger.
