# v0.20 Implementation Plan — Corpus-Wide Chunk Discovery

## Context

v0.19 shipped `find_chunk_parallels` as the **per-tablet probe** over sub-tablet granularity. The K.9508 ↔ K.5896 chunk_length=142 positive control, the K.3306 → K.6685 discovery (92.63% coverage), and the BM.77056 KAR-44 cross-curricular pattern across 12 sub-genres all came from per-tablet probing on the existing fuzzy 2-of-3 inverted-index infrastructure. No chunk-hash index was needed.

v0.20 ships the **corpus-wide enumeration tools** that DO need a chunk-hash index: `find_formulaic_passages`, `trace_chunk_diffusion`, `build_citation_graph`. These collapse the K.3306/BM.77056/Position-57 findings from "interesting one-off observations" into a primitive that surfaces *every* chunk-with-many-hosts in the corpus, *every* chunk's chronological diffusion, *every* commentary→base-text citation edge.

The binding constraint is **fragment-metadata coverage** (~0.6% default). Today's BM.77056 follow-up showed enrichment works mechanically (17/17 in 28 seconds). Rather than defer `build_citation_graph` to v0.21 because metadata is thin, v0.20 opens with a metadata-enrichment burst (v0.20.0-alpha, no code change, just API time), then ships all three tools together as a thematically unified Round-5 release.

## Phase 0 — Pre-v0.20 prerequisites (v0.19.1 work)

Two items must land before v0.20.0 begins. Both are short.

### 0.1 `host_genres_spanned` field on `find_chunk_parallels`

The BM.77056 cross-curricular finding (today's `b913d72`) exposed a structural limitation: `cross_genre_only=true` requires source genre, but eBL has no `genres[]` for some cluster seeds (notably BM.77056). Add a per-chunk field counting **distinct host genres** independently of source metadata. This is the right primitive for `find_formulaic_passages`' cross-curricular ranking, which won't depend on source metadata.

**Change in `src/chunkParallels.ts`** (~15 lines):
- Track `hostGenres: Set<string>` per chunk during aggregation
- Emit `host_genres_spanned: number` on the `ChunkParallel` output
- Update schema + tool description
- No semantic change to `cross_genre_count` — it stays as "count of hosts whose genre differs from source's genre"

### 0.2 K.3306 ↔ K.6685 scholarly review (Dane decision, blocking)

The v0.19 §3.6 amendment claim rests on K.3306 ↔ K.6685 being a real textual parallel. Risk: heavy `ABZ480 ABZ411` (= numeral 1) repetition in the chunks could mean this is calendrical/numerical coincidence, not real shared text. **Scholarly review must run before** `find_formulaic_passages` uses K.3306-style cases in its validation set, because a false-positive validation seed would bias the whole tool.

**Required from Dane:**
- Open the actual sign sequence in eBL for both K.3306 (positions 1–51 and 58–95) and K.6685 (full text)
- Determine: is this (a) true textual parallel, (b) calendrical/numerical formula coincidence, (c) unrecognized physical join, or (d) coincidental short formula?
- Document the verdict in `docs/v0.19-calibration-round4-chunk-parallels.md` under the Post-enrichment section
- If verdict is (b), retract the §3.6 amendment from the methods paper

## Phase 2 — v0.20.0 Day 1: chunk-hash index

### 2.1 `src/chunkIndex.ts` (new module, ~200 lines)

```ts
export type ChunkIndexEntry = {
  hash: string;               // sorted-trigram concatenation, length=20 window
  signs: string;              // reconstructed sign sequence (~22 signs)
  occurrences: Array<{
    tablet_id: string;
    start_position: number;   // in tablet's trigrams_ordered
  }>;
};

export type ChunkIndex = {
  version: string;            // built-against mcp version
  build_timestamp: string;
  total_tablets: number;
  total_windows_seen: number;
  total_unique_hashes: number;
  total_non_singleton_hashes: number; // post-pruning count
  entries: ChunkIndexEntry[];          // ranked by occurrences.length desc
};

export function loadChunkIndex(): ChunkIndex | null;
export function chunkIndexStats(): { loaded: boolean; entries: number; load_error: string | null };

// Query helpers (called by find_formulaic_passages / trace_chunk_diffusion / build_citation_graph)
export function getChunksAboveHostCount(min: number): ChunkIndexEntry[];
export function getChunksContaining(tabletId: string): ChunkIndexEntry[];
export function getChunkByHash(hash: string): ChunkIndexEntry | null;
```

**Storage:** single JSON file at `~/.cache/cuneiform-mcp/chunk-index.json`. Singleton pruning at build time (`occurrences.length === 1` entries dropped). Estimated size on 35K-tablet corpus: ~100–200 MB post-pruning (10× smaller than naive ~2 GB pre-pruning).

**Why single window length = 20:** the v0.19 calibration showed length=20 cleanly separates signal from noise. Shorter lengths (10, 15) admit numerical-formula and colophon-template false positives at scale. Incipit-targeted discovery (3–8 trigrams) is a different precision/recall regime; deferred to a future `find_incipits` tool with its own calibration.

**Why exact-hash (not fuzzy):** `find_chunk_parallels` already covers fuzzy per-tablet matching. The corpus-wide index needs to be cheap to query and exact-hash supports O(1) lookups. Fuzzy matching across the full corpus is a separate problem (deferred).

### 2.2 `scripts/build-chunk-index.mjs` (new build script)

Pattern-copy `scripts/build-anomaly-index.mjs`. Inputs: `~/.cache/cuneiform-mcp/all-signs-full.json`. Outputs: `~/.cache/cuneiform-mcp/chunk-index.json`. Runtime target: <15 minutes single-threaded on a 35K-tablet corpus.

Algorithm:
1. Load corpus via the existing `loadCorpus()` (or re-read all-signs-full.json directly)
2. For each tablet, slide a length-20 window over `trigrams_ordered`
3. For each window, compute hash = `trigrams_ordered.slice(i, i+20).join('|')`
4. Build inverted index: hash → Array<{tablet_id, start_position}>
5. Prune singletons (occurrences.length === 1)
6. Compute reconstructed sign sequence per surviving entry (use the same logic as `chunkParallels.ts:reconstructChunkSigns`)
7. Rank by occurrences.length desc
8. Write JSON

**Sanity check during build:** total non-singleton hashes should be in the 100K-500K range for the 35K-tablet corpus. If outside that range, investigate before shipping.

## Phase 4 — v0.20.0 Days 2–4: three new MCP tools

Built in parallel (matches the v0.18.x parallel-build cadence). Each tool is ~100–150 lines + registration in `src/index.ts`.

### 4.1 `find_formulaic_passages` (Day 2)

```ts
findFormulaicPassages({
  min_hosts?: number = 20,         // chunks shared with ≥ N tablets
  top_k?: number = 50,
  cross_genre_only?: boolean,      // require host_genres_spanned ≥ 2
  cross_period_only?: boolean,
  exclude_prefixes?: string[],     // e.g. ['Asb.'] to skip colophon prototypes
});
// Returns:
{
  passages: Array<{
    chunk_hash: string,
    chunk_signs: string,
    chunk_length: number,
    host_count: number,
    host_tablets: Array<{ tablet_id, period?, genre? }>,
    host_genres_spanned: number,
    host_periods_spanned: number,
    novelty_score: number,         // host_genres_spanned × log(host_count) — rewards cross-curricular formulae
  }>,
  index_stats: { ... },
}
```

**Direct backbone:** queries the chunk-hash index for `entries.filter(e => e.occurrences.length >= min_hosts)`, ranks by `host_genres_spanned * log(occurrences.length)`. Joins fragment-metadata for period/genre attribution.

**Validation case:** BM.77056 position-57 chunks should surface in the top-N (those were the v0.19 cross-curricular finding — three different chunk lengths at the same position with hosts spanning six sub-genres). Negative case: pure colophon templates (`Asb.*` prototypes) should rank low because their host-genre diversity is uniform (all Library of Ashurbanipal).

### 4.2 `trace_chunk_diffusion` (Day 3)

```ts
traceChunkDiffusion({
  chunk_hash?: string,             // exact chunk to trace
  tablet_id?: string,              // OR: pick a chunk from this tablet
  chunk_index?: number,            // (with tablet_id) which of the tablet's chunks
});
// Returns:
{
  chunk_hash: string,
  chunk_signs: string,
  diffusion: Array<{
    period: string | null,
    period_sort_key: number,
    tablets: Array<{ tablet_id, genre? }>,
  }>,                              // ordered by period_sort_key ascending
  earliest_period: string,
  latest_period: string,
  period_span_years_approx: number,
  cross_period_count: number,
}
```

**Direct backbone:** queries chunk-hash index for the chunk, looks up `getPeriodInfo` for each host, groups by period, sorts diffusion array by sort_key.

**Validation case:** trace a known KAR-44 incipit chunk (whatever surfaces from `find_formulaic_passages`) → should show period diffusion across Old Babylonian → Middle Babylonian → Neo-Assyrian → Neo-Babylonian → Hellenistic, mirroring the documented transmission history of the *āšipūtu* curriculum.

### 4.3 `build_citation_graph` (Day 4)

```ts
buildCitationGraph({
  min_shared_chunks?: number = 2,   // require ≥ N chunks shared per edge
  commentary_genres?: string[],      // default ['Commentary', 'Commentaries']
  top_k_edges?: number = 100,
});
// Returns:
{
  edges: Array<{
    cited_by: string,                // commentary tablet ID
    cited_by_genre: string,
    cites: string,                   // base-text tablet ID
    cites_genre: string,
    shared_chunks: Array<{ hash, length, signs }>,
    edge_weight: number,             // sum of chunk lengths
  }>,
  index_stats: {
    candidates_examined: number,
    edges_after_filter: number,
    metadata_coverage_pct: number,   // critical context: % of corpus with genre attribution
  },
}
```

**Direct backbone:** for each chunk in the chunk-hash index, partition its hosts into `commentary_genre_hosts` and `base_text_hosts` using `getPrimaryGenre`. If both partitions non-empty, every (commentary, base) pair gets +1 to its edge weight. Singleton edges (single chunk shared) filtered by `min_shared_chunks`.

**Validation case:** BM.47463 ↔ CBS.6060 (the v0.18.19 `commentary_quotes_base_text` verdict pair) should appear as an edge — Šurpu commentary citing Šurpu base text, 147-sign shared chain.

## Phase 5 — v0.20.0 Day 5: Round-5 calibration audit

`scripts/round5-corpus-wide-chunk-audit.mjs` (pattern-copy v0.19 round-4). Five tests:

1. **Index build sanity** — `chunk_index.json` exists, non-singleton hash count in 100K–500K range, sample 10 hashes and verify their occurrences reconstruct correct signs.

2. **`find_formulaic_passages` positive control** — top-20 should include the BM.77056 position-57 chunk (length 25, hosts spanning Mīs pî + Ritual texts genres). If not present in top-50, debug.

3. **`find_formulaic_passages` negative control** — `Asb.c` / `Asb.d` colophon prototypes should rank LOW on `novelty_score` (high host count but low genre diversity → score multiplier = 1, not boosted). If they top the leaderboard, the novelty scoring is wrong.

4. **`trace_chunk_diffusion` positive control** — pick the highest-host-count chunk from Test 2, verify the diffusion array spans at least two periods.

5. **`build_citation_graph` positive control** — BM.47463 ↔ CBS.6060 must appear as an edge with ≥1 shared chunk. If absent: either chunk-hash index missed the 147-sign Šurpu chain (build bug) or the commentary-vs-base partitioning is broken.

**No 20-tablet random sample** in Round 5 — that pattern is for per-tablet probes (v0.18.19, v0.19). Corpus-wide tools are validated against known anchor cases instead.

## Phase 6 — Methods paper §3.10 + claims 23–25

**`docs/methods-paper-cdlj-submission.md`** new section after §3.9.1 (current end of §3.9):

### §3.10 Corpus-Wide Chunk Discovery (v0.20.0)

23. **`[my synthesis]`** **The chunk-hash index transforms sub-tablet discovery from per-tablet probe to corpus-wide enumeration without ML primitives.** The same trigram alignment that produces `find_chunk_parallels`' per-tablet output, applied at corpus scale with singleton pruning, surfaces every passage shared between 2+ tablets in milliseconds. No embedding-space, no learned model — just exact hashing.

24. **`[my synthesis]`** **Formulaic-passage discovery recovers the KAR-44 curriculum's most-canonical incipits as the highest-host-count chunks.** This is the third independent recovery of the curriculum from the same corpus: v0.17 whole-tablet clustering (§3.1), v0.19 chunk-parallels per-tablet probe (§3.9.1), v0.20 corpus-wide chunk-hash enumeration (§3.10). Three orthogonal methods converging on the same structure is strong evidence the curriculum recovery is methodology-independent.

25. **`[my synthesis]`** **The citation graph derived from chunks shared between commentary and base-text genre hosts is a corpus-level structural primitive, not a pair-level diagnostic.** The v0.18.19 `commentary_quotes_base_text` verdict in `compare_tablet_pair` answered "is THIS pair a commentary/base relationship?". `build_citation_graph` answers "what does the WHOLE corpus's quotation network look like?" — and PageRank-style centrality on the graph identifies canonical-source tablets without scholar curation.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Metadata enrichment burst hits eBL rate limits | Run after-hours, polite 5-concurrency, retry on 429 |
| Chunk-index build OOMs on full corpus | Stream-process: build hash→occurrences map per-tablet, write JSON in streaming mode |
| K.3306↔K.6685 verdict is "calendrical coincidence" → §3.6 amendment retraction | Mitigate before v0.20.0 ships — Phase 0.2 is blocking |
| `find_formulaic_passages` colophon templates dominate (false positives) | `exclude_prefixes` param + novelty score weighted by genre diversity (not raw host count) |
| `build_citation_graph` edges are dominated by within-Library-of-Ashurbanipal noise (not real citation) | `min_shared_chunks ≥ 2` floor + cross-genre filter (commentary vs base) |
| Length=20 misses short incipits (3–8 sign formulae) | Defer to v0.21+ `find_incipits` tool with length=10 index; document the limitation in §3.10 |

## Out of scope for v0.20 (queued for v0.21+)

- **Incipit-targeted discovery** (`find_incipits`) — needs length=10 chunk-hash index, different precision regime.
- **Cross-corpus comparative** (Hebrew Bible / Ugaritic / Hittite per methods paper §5.4) — needs a second corpus integration.
- **Active-learning prioritization** by chunk-coverage gaps — needs the bi-orphan validation backlog as a queue, separate UX problem.
- **Per-archetype threshold matrix** (Lever 5 deferred from Round 3) — still deferred; not blocking v0.20.
- **Sign-form variant normalization at query time** — methods paper §5.4 future work, separate analytical primitive.

The v0.20 chunk-hash index doesn't address these; they remain on the roadmap.
