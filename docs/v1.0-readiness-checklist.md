### 2. API freeze decisions (Dane judgment required)

For each of the 74 tools, decide:
- **Stable surface (v1.0 freezes signature)** — most tools, probably
- **Experimental surface (v1.0 marks as "may change")** — recent additions: v0.25 ensemble, v0.26 per-period
- **Deprecated** — anything superseded by v0.19+ should be marked

Specific questions:
- Should `find_chunk_parallels` (v0.19, fuzzy per-tablet) and `find_formulaic_passages` (v0.20, exact corpus-wide) consolidate into one tool with a `mode` parameter, or stay as parallel tools?
- Should the sign2vec axis stay as 3 tools (single-config, ensemble, per-period) or consolidate?
- Should the legacy v0.5-v0.13 tools (lookup_sign, search_oracc, etc.) be marked stable or deprecated in favor of the v0.18+ tools?

### 3. Cross-axis Bayesian fusion

The original v0.20 roadmap proposed `compute_joint_pair_score(A, B)` aggregating all axes into one probabilistic verdict. Needs:
- **Gold-standard labeled pair set** (~50-100 manually-validated positives + negatives) — currently 0
- Logistic regression or simple Bayesian model on the per-pair feature vector
- Round-N audit + methods paper §3.15

**Action item for Dane:** decide whether to spend time labeling pairs or to defer Bayesian fusion to post-v1.0.

The v0.25 substitution-lift z-score (+2.24σ separation on K.5896 ↔ K.9508) is the empirical proof-of-concept that single-axis baseline-normalization works. Bayesian fusion would extend this to multi-axis. But it requires labels.

## Recommended sequencing toward v1.0

1. **Wait on Jiménez reply** (rate-limit + Auth0 + endorsement). 1-3 weeks expected silence.
2. **JOHD submission prep** if Jiménez can't help arXiv-endorse: write JOHD-format cover letter, ensure paper formatting matches their template.
3. **Labeled-pair collection** (parallel to #1-#2): set aside 2-3 hours to manually classify 50 pairs as sibling/non-sibling. Even a small gold-standard set unblocks Bayesian fusion.
4. **API freeze pass**: 1-day review of all 74 tools against the v1.0 stable/experimental/deprecated classification.
5. **PROTOCOL.md + README.md refresh**: ~half day.
6. **CI: GitHub Action running smoke + regression audit on every push to main.** ~2 hours.
7. **Acceptance signal**: paper accepted somewhere → declare v1.0 + freeze.

Estimated total work from Dane's side: ~1 working week distributed across whichever days the methods-paper venue responses come in.
