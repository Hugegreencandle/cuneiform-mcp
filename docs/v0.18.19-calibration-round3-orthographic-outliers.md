# v0.18.19 Calibration Audit — Round 3, Untested-Tool: `find_orthographic_outliers_in_prefix`

*v0.18.10 shipped `find_orthographic_outliers_in_prefix` without a dedicated audit. Round-3 plan flagged the risk that "outliers may be sign-count artifacts" — small-signature tablets have noisier LLR profiles and might surface as outliers due to statistical noise rather than orthographic difference. **NO-OP. The tool is working correctly. The default `min_sign_count` threshold is effectively dead code (sort+cap dominates the cohort definition) but this is a documentation footnote, not a calibration ship.***

---

## Hypothesis

`min_sign_count` default is 50. If small-signature tablets dominate the outlier surface — i.e., outliers' median sign_count is well below the cohort median — that's a precision leak. Fix candidate: tighten `min_sign_count` default to 100 or 150.

---

## Empirical decomposition

### K prefix (`min_sign_count=50`, `max_tablets_to_scan=500`)

```
Cohort size: 500
Mean cosine to centroid: 0.3332
Median: 0.3236
Stdev: 0.1027
Most typical: K.2903 (0.62), K.12068 (0.61), K.2300 (0.60)

Top-20 outliers — sign_count distribution:
  min=588, median=934, max=2352
```

**Outlier sign_count median is 934 — well above the 50-sign default threshold.** No small-tablet leak. The lowest sign_count among outliers (588) is itself a substantial transliterated tablet.

### BM prefix (same params)

```
Cohort size: 500
Mean cosine to centroid: 0.4298
Median: 0.3220
Stdev: 0.3176  (3× higher than K — bimodal distribution)
Most typical: BM.34926 (0.92), BM.35086 (0.91), BM.43025 (0.91)

Top-20 outliers — sign_count distribution:
  min=541, median=775, max=2548
```

**Outlier sign_count median is 775 — well above the 50-sign threshold.** BM's bimodal cosine distribution (mean 0.43 + stdev 0.32 → wide spread) is itself a finding: BM has a tight modal cohort + a long tail of orthographically distinct tablets. The tool correctly surfaces this.

### Threshold sweep — `min_sign_count` is effectively dead code

Sweeping the K prefix `min_sign_count` across the audit grid:

| min_sign_count | Cohort size | Top-1 outlier | Outlier sign_count min/median |
|---|---|---|---|
| 50  | 500 | K.4395 (sc=678, cos=0.058) | 673 / 934 |
| 80  | 500 | K.4395 (sc=678, cos=0.058) | 673 / 934 |
| 100 | 500 | K.4395 (sc=678, cos=0.058) | 673 / 934 |
| 150 | 500 | K.4395 (sc=678, cos=0.058) | 673 / 934 |
| 200 | 500 | K.4395 (sc=678, cos=0.058) | 673 / 934 |

**Identical outliers across all threshold values.** The default 50-sign threshold doesn't bind because `max_tablets_to_scan=500` + sort-by-sign_count-desc fills the cohort with the 500 largest tablets regardless. Even at `min_sign_count=200`, the binding constraint is "top 500 by sign_count", not the threshold.

---

## Verdict — NO-OP

The tool is correct. Outliers are genuinely scribal-signature outliers, not sign-count artifacts. The threshold sweep shows the default `min_sign_count=50` is harmless but effectively dead code.

| Hypothesis check | Result |
|---|---|
| Small-tablet leak in outlier surface | ✗ Outlier median sign_count is 934 (K) / 775 (BM), well above the 50-sign default |
| Threshold tightening would improve precision | ✗ Identical outliers across min_sign_count ∈ {50, 80, 100, 150, 200} |
| K vs BM differential is a calibration artifact | ✗ K has homogeneous cohort (stdev 0.10), BM is bimodal (stdev 0.32) — real corpus structure |

**No fix shipped. The tool stays calibrated.**

---

## Documentation footnote (not a fix)

The default `min_sign_count=50` is harmless but redundant. Future-self could tighten `max_tablets_to_scan` to a lower number (say 200) to expose `min_sign_count` as a binding constraint, or note in the tool description that "the cohort is always the top-N largest tablets in the prefix; `min_sign_count` only filters tablets below that absolute threshold, which the sort cap usually subsumes." This is a docs issue, not a calibration ship.
