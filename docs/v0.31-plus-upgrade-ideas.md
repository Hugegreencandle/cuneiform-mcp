# cuneiform-mcp v0.31+ Upgrade Ideas (for Ultraplan refinement)

Draft idea-list for the next round of upgrades, intended as input to `/ultraplan`. Each idea includes a one-line capability, motivating case, rough effort, and methods-paper relevance.

The v0.18 → v0.30 arc shipped 81 tools across 17 calibration rounds. v1.0 is gated on (a) paper acceptance, (b) ≥100 labeled pairs, (c) API-freeze decisions. The ideas below are CANDIDATE work for the v0.31+ pre-v1.0 window — pick whichever subset survives the refinement pass.

## Tier 1 — High novelty, single-tool scope

### 1. `identify_composition` — composition assignment for a query tablet

Given a tablet's signs, return the most-probable composition (Mīs pî, Šurpu, EAE, Šumma izbu, etc.) via joint inference across (a) per-period chunk-hash overlap with known composition exemplars, (b) sign2vec semantic match against exemplar centroids, (c) scribal-school membership. Returns ranked candidate compositions + confidence. ~3 days. Methods §3.18.

### 2. `build_stemma_with_rooting` — rooted recension trees

v0.22 `build_canonical_recension_tree` produces UNROOTED neighbor-joining trees. Add rooting heuristics: (a) earliest-period as root, (b) chunk-coverage-centroid as root, (c) external-witness as root (an Old Babylonian forerunner if present). Outputs a rooted Newick + root-choice justification. ~3 days.

### 3. `find_composition_lineage` — composition diffusion through scribal schools

Compose v0.22 scribal-school graph + v0.22 stemma + per-period chunks: for a composition (seed tablet), trace which scribal schools transmitted it across periods. "The Mīs pî was copied by Sippar atelier 4 in Neo-Babylonian, then re-copied by Nineveh EAE atelier in Neo-Assyrian via [intermediate witness]." ~5 days. Methods §3.18.x.

### 4. `score_tablet_completeness` — fragment-vs-composition gap

Given a fragment, estimate what fraction of the original composition is preserved. Uses chunk-coverage against the composition's most-complete witness + lacuna-density per the v0.18.0 / v0.30 lacuna tools. Useful for prioritizing reconstruction effort. ~2 days.

### 5. `feedback_loop_for_validation_queue` — active learning

Extend v0.21 `prioritize_validation_queue` with persistent resolution feedback. User marks a candidate as "real sibling" or "false positive"; the resolution feeds back into the v0.29 Bayesian model as a NEW labeled pair, growing the labeled set toward the ≥100 v1.0 threshold organically. ~3 days. **Closes one of the v1.0 gates over time.**

## Tier 2 — Cross-tool integrations

### 6. `compare_clusters_v2` — multi-axis cluster comparison with archetype awareness

v0.18 `compare_clusters` predates the v0.26 archetype matrix. Re-implement using per-archetype thresholds — each side of the cluster pair gets its own optimal calibration profile before comparison. ~3 days.

### 7. `damaged_passage_composition_probability`

Given a damaged passage, return the probability distribution over candidate compositions. Combines v0.30 lacuna sign2vec + v0.20 chunk-hash + v0.29 Bayesian fusion features. Aggregates across plausible sign restorations. ~5 days.

### 8. `bayesian_fusion_at_scale` — V1.0 production model

Train the v0.29 model on the resolved-validation-queue labels (depends on #5). Production-quality cross-axis fusion. ~2 days once labels exist. **Closes the second v1.0 gate.**

## Tier 3 — New data axes

### 9. `find_lemma_parallel` — lemma-aware textual parallels

eBL provides lemma/translation data on transliterations (currently unused). Extract lemmas for a sample of high-host tablets and build a lemma-overlap parallel finder, complementary to the v0.18 sign-trigram approach. ~5 days + enrichment burst.

### 10. `extract_citation_network` — bibliography mining

eBL has a bibliography field per fragment. Build a citation network from published-scholarship references. Useful for: (a) auto-citing methods paper findings, (b) bibliography auto-completion, (c) provenance tracking. ~3 days + bibliography-data enrichment burst.

### 11. `find_sign_glyph` — Unicode glyph lookup

Convert ABZ codes to Unicode cuneiform glyphs (𒀭, 𒈗, etc.) for display in tool outputs. Today the tools output `ABZ480 ABZ411 …` strings; researchers would prefer to see the actual cuneiform when possible. Pure data-loading task, no novel methodology. ~1 day.

## Tier 4 — Documentation + ops

### 12. PROTOCOL.md refresh

Verify the v0.5 envelope spec matches v0.18+ actual outputs. May surface drift. ~half day.

### 13. CI: GitHub Action running smoke + regression audit on push to main

Catches regressions before they reach main. ~2 hours. **Closes one of the v1.0-readiness items.**

### 14. API freeze audit

For each of the 81 tools, classify stable / experimental / deprecated. Generate a `docs/API-STABILITY-v1.0.md` map. Requires Dane judgment on edge cases (e.g., are the 3 sign2vec variants — single-config / ensemble / per-period — all stable, or should the ensemble + per-period get the experimental tag?). ~1 day collaborative.

## Tier 5 — Post-v1.0 / speculative

### 15. Cross-corpus integration (Hebrew Bible / Ugaritic / Hittite)

Methods paper §5.4 future work. Tokenize external corpora into the same trigram + sign2vec frameworks. Massive scope; defer until v1.0.

### 16. Hosted MCP deployment (Cloudflare Workers + R2)

Per `docs/v1.0-cloudflare-hosting-plan.md`. Gated on paper acceptance per the existing plan. ~1-2 weeks once triggered.

### 17. v0.27 deep-dive — register-matched per-archetype embeddings

The next conditional-calibration axis: train embeddings on (period × register × archetype) sub-corpora. Currently we have period × register (v0.27). Adding archetype would resolve some of the v0.27 register-matched noise. ~1 week, methodologically rich but specific.

### 18. Bibliography-aware Bayesian fusion

Add bibliography-overlap as a 6th axis in v0.29's Bayesian fusion. Two tablets cited in the same publication are more likely related. Requires #10 + retraining. ~3 days after #10.

## What's deliberately NOT in this list

- Tools that duplicate existing capability (we have 81 tools; resist sprawl)
- Anything requiring Dane's judgment that I shouldn't pre-decide (API freeze edge cases, paper revisions)
- Speculative ML applications without clear empirical hypotheses (sign-detection from images, LLM fine-tuning) — separate research arcs
- Anything that would change the methods paper's load-bearing claims (the §3.x edifice is published-ready; new claims extend, don't rewrite)
